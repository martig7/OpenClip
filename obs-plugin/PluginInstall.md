# OBS Plugin Installation Reference

## Overview

OBS loads plugins by scanning specific directories on startup. Installers only
need to copy files to the right place — OBS handles discovery automatically.

## Plugin File Structure

Each plugin consists of two parts:

- **Binary:** the compiled `.dll` (Windows), `.so` (Linux), or `.dylib` (macOS)
- **Data:** locale/resource files (e.g. `en-US.ini`)

### Directory Layout (Windows)

**System-wide (requires admin):**

```text
C:\Program Files\obs-studio\obs-plugins\64bit\PluginName.dll
C:\Program Files\obs-studio\data\obs-plugins\PluginName\
```

**System-wide (preferred, no admin, OBS v28+):**

```text
C:\ProgramData\obs-studio\plugins\PluginName\obs-plugins\64bit\PluginName.dll
C:\ProgramData\obs-studio\plugins\PluginName\data\obs-plugins\PluginName\
```

**User-only:**

```text
%AppData%\obs-studio\plugins\PluginName\obs-plugins\64bit\PluginName.dll
%AppData%\obs-studio\plugins\PluginName\data\obs-plugins\PluginName\
```

## Installation

### What an installer does

1. Copies `.dll` and data files into the appropriate directory (see above)
2. Registers an uninstall entry in the Windows registry:
   `HKLM\Software\Microsoft\Windows\CurrentVersion\Uninstall\PluginName`
3. Does **not** touch `modules.json` — OBS handles that itself on next launch

### Standard toolchain (obs-plugintemplate)

The official OBS plugin template (https://github.com/obsproject/obs-plugintemplate)
is the de facto standard for building and packaging plugins. It uses:

- **CMake** with `CMakePresets.json` for cross-platform builds
- **`buildspec.json`** to declare plugin metadata and dependency versions
- **GitHub Actions** to automatically build and package on tagged releases:
  - Windows: InnoSetup `.exe`
  - macOS: signed/notarized `.pkg`
  - Linux: `.deb`, `.rpm`, or tarball

The packaging flow is:

```text
CMake build → CMake install → platform packager → GitHub Actions releases on tag push
```

## modules.json

### Location

```text
%AppData%\Roaming\obs-studio\plugin_manager\modules.json
```

### Purpose

Introduced in OBS v32. OBS reads this file to track which plugins it has
discovered and whether they are enabled or disabled in the Plugin Manager UI.

### Behavior

- **On install:** OBS auto-discovers the new plugin on next launch and adds it
  to `modules.json`. Installers do not need to write to this file.
- **On uninstall:** OBS does not auto-clean stale entries. If plugin files are
  removed without OBS running, the entry remains in `modules.json` and appears
  greyed out in the Plugin Manager.

### Agent guidance

- **Program Files install** (`obs-plugins/64bit/`): OBS auto-discovers on next
  launch and adds the entry itself. Do not write `modules.json`.
- **AppData/ProgramData install** (per-user plugin path): OBS does NOT
  auto-scan these paths for discovery — it only loads from them if the
  `module_name` entry already exists in `modules.json`. You must write the
  entry yourself with all 9 required fields on install.
- If removing a plugin programmatically, delete the plugin's entry from
  `modules.json` to avoid stale UI state. The file is standard JSON, so this
  is straightforward to do with any JSON-capable tool.
- Always make changes to `modules.json` while OBS is not running.

## Troubleshooting

### Plugin shows "[PLUGIN NOT FOUND]" in the OBS Plugin Manager

**Symptom:** The plugin appears in the OBS Plugin Manager with a "[PLUGIN NOT
FOUND]" status. The DLL exists on disk (confirmed via file explorer and
PowerShell), loads correctly when tested manually with `LoadLibraryW`, exports
`obs_module_load`, and has correct file permissions. No Zone.Identifier
alternate data stream. OBS logs contain no mention of the plugin — not even a
"Skipping module" entry.

EVERYTHING BELOW HERE IS HIGHLY DUBIOUS AND SHOULD NOT BE IMMEDIATELY TRUSTED.
modules.json is important, but it gets autogenerated. 
Something else is causing the issue. Suggest a reinstall to fix for now.

**Root cause:** OBS v32 introduced a persistent plugin manager state in
`modules.json`. If a previous installation attempt left a stale or malformed
entry in this file (e.g., with an incorrect `module_path` that no longer matches
the actual install location), OBS will show "[PLUGIN NOT FOUND]" for that entry.
Because OBS reads the cached entry instead of re-scanning, it never attempts to
load the DLL from disk — so no log output is produced. The file-level tests
(permissions, exports, `LoadLibraryW`) all pass because the problem is in OBS's
state tracking layer, not the DLL itself.

**Fix:** Reinstall OBS. This resets `modules.json` and all plugin manager state,
allowing OBS to cleanly rediscover all installed plugins on next launch.
```
**Alternative (without reinstalling OBS):** Manually delete or correct the
plugin's entry in `modules.json` while OBS is closed:

```text
%AppData%\Roaming\obs-studio\plugin_manager\modules.json
```

Remove the stale entry for the plugin, then relaunch OBS. It will re-scan and
add a fresh, correct entry.```
